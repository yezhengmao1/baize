# CLAUDE.md

Working directory for the **nvidia-nsys-analyzer** agent. Python tools for analyzing NVIDIA Nsight Systems profiling data (`.nsys-rep` ŌåÆ SQLite).

**Status**: Installable package `baize` with six console entry points ŌĆö `gpu-flame` (per-step kernelŌåÆNVTX flame graph, incl. `--diff`), `gpu-comm` (cross-rank communication ŌĆö NCCL collectives/P2P + DeepEP MoE all-to-all: volume / busbw / skew), `gpu-shape` (per-operator input-shape table, compute only), `gpu-exporter` (Chrome/Perfetto trace JSON), `gpu-groups` (Megatron parallel-group resolver, config-only ŌĆö no profile), and `gpu-sched` (config-only op-level 1F1B pipeline timeline SVG ŌĆö wall-clock-scaled, sub-phase-decomposed). (The earlier suite `perf_stat.py`, `kernel_analysis.py`, `comm_bw.py`, ŌĆ” and the standalone `report.py`/`kernel_inspect.py`/`comm_inspect.py` were removed.)

**Data model rule**: kernels are the primary entity; NVTX context is attached via the kernel->NVTX mapping (`utils.nvtx.NvtxIndex`, or `utils.comm` for the NCCL-kernelŌåÆop join). Tools may reverse-look-up kernels through this mapping but must not query `NVTX_EVENTS` as an independent first-class table.

## Environment

- **nsys**: `/usr/local/bin/nsys` (2026.2.1); convert with `nsys export --type sqlite -o out.sqlite in.nsys-rep`.
- **Python**: 3.12. Runtime deps: `numpy`, `matplotlib` (declared in `pyproject.toml`); `sqlite3` is stdlib.

## Install

```bash
pip install -e .          # from repo root; provides gpu-flame / gpu-comm / gpu-shape / gpu-exporter / gpu-groups / gpu-sched
```

Entry points run the tools' `if __name__ == "__main__":` blocks via small shims in `nsys_tools/cli.py` (so the CLI modules keep that style). Equivalent: `python -m nsys_tools.tools.flamegraph ŌĆ”`.

## Tools

| Binary | Purpose | Usage |
|---|---|---|
| `gpu-flame` | Step-boundary detector + optional flame graph. Detects step boundaries from the kernel->NVTX mapping (`--step-nvtx SUBSTR`, default `Optimizer.step`), skips warmup (`--skip-steps N`, default 1), prints the per-step window table. With `--flamegraph OUT` builds an interactive HTML flame graph (`<OUT>.html`, d3-flame-graph) with **four charts over one shared hierarchy**: **Solo** (non-overlap; widths = ╬Ż per-activity solo time / `active_count==1`, the portion blocking the GPU clock), **Sum** (overlap; widths = ╬Ż GPU-activity time, color saturation encodes overlap: vivid = exposed, pale = hidden), **Count** (widths = number of GPU activities / kernel launches ŌĆö wide here but narrow in Sum ŌćÆ launch-bound, a fusion/CUDA-graph target), and **Sum + idle** (the Sum chart with GPU-idle added). `sumŌåÆsolo` shrink = overlapped/hidden, equal = exposed; every node carries `sum`/`solo`/`count` (StackNode.count) and the tooltip shows time (sum/solo/overlap%) + count/% of launches. `--stack-depth N` truncates the CPU NVTX path. Leaves are compute kernels **plus GPU memcpy (H2D/D2H/D2D) and memset** (`--include-memcpy`/`--no-include-memcpy`, default on) ŌĆö these live in the separate `CUPTI_ACTIVITY_KIND_MEMCPY`/`_MEMSET` tables (joined to their launch API via `correlationId`, same NVTX mapping as kernels), colored green/purple; with them off, memcpy-only spans fall into idle. **GPU idle appears only in the third chart**: gaps are attributed to the NVTX scope enclosing each gap (`--attribute-idle`/`--no-attribute-idle`, default on) ŌĆö magenta `<idle>` leaves so you can see *which phase* the GPU stalled in (gaps resolved via the same sweep-line as kernels, keyed on GPU wall-time ŌåÆ approximate under heavy CPU-ahead async); `--no-attribute-idle` makes it one root block. The Solo/Sum charts are pure GPU activity (no idle). Data model: `load_flame_tree` returns `(tree, tree_idle)` ŌĆö `tree_idle` is `tree.clone()` + idle; `write_html` emits both as `__DATA__`/`__DATA_IDLE__`. **`--diff BASELINE_DB`** emits a differential flame graph instead: `db` is the "after", BASELINE the "before", values per-step-normalized. It renders **the whole 4-view flame graph tripled** ŌĆö **Before** (baseline) / **After** (current) / **Diff**, each showing all four views (Solo / Sum / Count / Sum+idle) = 12 charts over one merged hierarchy. Before/After use the normal color-by-kind (Sum keeps overlap saturation); the **Diff** row has the After widths but colors each frame by that view's delta (red = grew, blue = shrank). Merged tree carries both sides' `{sum,solo,count}` per node (+ an idle-tree for Sum+idle); renderer: `write_diff_html` + `_merge_metrics_json` + `templates/flamegraph_diff.html`. | `gpu-flame <db.sqlite> [--step-nvtx SUBSTR] [--skip-steps N] [--stack-depth N] [--no-include-memcpy] [--no-attribute-idle] [--flamegraph OUT] [--diff BASELINE_DB]` |
| `gpu-shape` | Per-operator input-shape table over the post-warmup window, **compute only** (communication excluded). Reuses `gpu-flame`'s step detector (`--step-nvtx`/`--skip-steps`) to pick the window, reads every kernel via `KERNEL_SQL`, and attributes each to its **innermost enclosing aten-op NVTX frame** ŌĆö the frame carrying a `sizes = [[...]]` shape annotation (torch `emit_nvtx(record_shapes=True)`), resolved through the kernelŌåÆNVTX-stack mapping. `op_frame` skips comm-plumbing frames (`nccl`/`c10d::`/`record_param_comms`/`NcclGroup`/`deep_ep`/`Fused{Dispatch,Combine}`) and `ncclDevKernel*` kernels are dropped outright, so the table is pure compute. One op **invocation** = one op-frame instance `(op, shapes, start, end)`; instances roll up per **(operator, input-shapes)** ŌåÆ `calls` (invocations), `kern` (kernels), ╬Ż`gpu` time, `avg/call`, `%` of compute GPU time. Header reports excluded comm / no-shape kernel counts+time so coverage is explicit. `--sort {time,calls,op}`, `--top N`, `--csv OUT` (full untruncated rows). **`--html OUT`** writes a self-contained interactive page (`write_shape_html` + `templates/shape_table.html`, no CDN; styled to match `flamegraph.html`): a summary note (compute ╬Ż / #ops / kernels / excluded comm & no-shape), ranked horizontal bars of the top time sinks and a sortable/filterable table, both driven by a shared grouping toggle ŌĆö **by (op,shapes)** / **by operator** / **by kernel** ŌĆö all compute-only. **by kernel** aggregates every compute kernel by `shortName` (`build_shape_table` also returns `kern_names`; **all communication excluded** ŌĆö NCCL by name via `_COMM_KERNEL_PREFIX`, DeepEP `dispatch`/`combine` etc. by their enclosing comm NVTX scope via `_stack_has_comm`), including the kernels the op table drops as no-shape; its table columns switch to launches/╬Żgpu/avg/min/max. Each bar colored by name via the same fnv1a-hashŌåÆHSL scheme as the flame graph; hover = full shapes / per-call & min-max stats. | `gpu-shape <db.sqlite> [--step-nvtx SUBSTR] [--skip-steps N] [--sort {time,calls,op}] [--top N] [--csv OUT] [--html OUT]` |
| `gpu-comm` | Cross-rank communication report over many ranks. Loads each rank's comm kernels **in parallel** (`--jobs N`, multiprocessing; ranks are independent files). Every comm row is keyed by its **issuing NVTX scope** ŌĆö the model phase / call site that issued it (e.g. `_LinearBackward`, `SinkCorrectionWithCPFunc`, or a bare `c10d::allreduce_coalesced_` / `c10d::send` when no model module wraps it), resolved via the kernelŌåÆNVTX-stack mapping and **falling back to the outermost frame** when every enclosing frame is comm plumbing (`nccl`/`c10d::`/`record_param_comms`/`NcclGroup`). Three sections: **(1) Collectives** (`ncclDevKernel` AllGather/ReduceScatter/AllReduce/Broadcast): aligned across ranks by **(comm-handle, op, seq)** ŌĆö the `comm=0xŌĆ”` handle is the real PG identity (globally consistent across the ranks sharing the group), so each collective instance's per-rank durations and true group size are recovered exactly (no traffic-signature heuristic), then rolled up per (scope, op, width) ŌåÆ `nranks` (real group width, e.g. 8/32/96), per-rank `calls`, ╬Ż`vol/rank` (per rank, ├Ę group count), **busbw@floor** (`factor(op,nranks)├Śbytes / min-time-across-ranks`, busy-wait removed ŌåÆ achievable BW), **wait%** (`╬Ż(timeŌłÆfloor)/╬Żtime`, floor = min within each collective instance) + worst rank. **(2) P2P send/recv**: peer-based, no `comm=`/seq ŌåÆ no clean group, so only group-independent quantities per (scope, op): ╬Ż`vol/rank`, `calls/rank`, per-rank GPU-time floor..max. **(3) EP all-to-all (DeepEP)** (`deep_ep::` dispatch/combine, incl. notify/layout helpers; volume from `FusedDispatch`/`FusedCombine` NVTX sizes): per (scope, phase), skew computed **within EP groups** of `--ep N` consecutive ranks (`rank//N`, default 8 = intranode) ŌĆö a flat all-ranks floor would conflate the independent groups. `--p2p-dtype-bytes N` sizes the marker-less P2P/DeepEP volume (default 2 = bf16; 1 = fp8, 4 = fp32). `--heatmap OUT` writes a per-rank├Ścomm skew PNG (`--by-node` averages rows by node; `--sort {slowness,id}`). | `gpu-comm <db.sqlite>... [--step-nvtx SUBSTR] [--skip-steps N] [--p2p-dtype-bytes N] [--ep N] [--heatmap OUT] [--by-node] [--sort {slowness,id}] [--jobs N]` |
| `gpu-exporter` | Export a step (or the whole post-warmup window) as **Chrome Trace Event Format JSON** ŌĆö what `chrome://tracing` and the Perfetto UI ingest (nsys itself has no such export). Reuses `gpu-flame`'s step detector (`--step-nvtx`/`--skip-steps`; `--step N` picks one 1-based step, else the full post-warmup window). Default **split view**: GPU streams carry kernels/memcpy/memset, NVTX ranges stay on the CPU threads that issued them; `--project` instead resolves each GPU op's enclosing NVTX stack (via the kernelŌåÆNVTX launch-API mapping) and nests it onto the op's GPU stream so every kernel sits inside its issuing scopes (kernel = leaf), `--stack-depth N` caps that nesting. Multiple ranks merge into one trace (each rank its own pid namespace, `R<rank>` track prefix) **wall-clock aligned by each file's session-start UTC epoch** so cross-node ranks land on one real timeline and skew is visible (`--no-align` overlays at t=0). Extras: `--cpu-nvtx`/`--no-cpu-nvtx`, **`--cuda-api`/`--no-cuda-api`** (CUDA runtime API ŌĆö `cudaLaunchKernel`, ŌĆ” ŌĆö on CPU threads, **default on**, heavy), **`--flows`/`--no-flows`** (CPU-launchŌåÆGPU-kernel flow arrows, **default on** ŌĆö click a GPU kernel in Perfetto to jump to its launch site; both endpoints use `bp:"e"` to bind the enclosing slice ŌĆö the `cudaLaunchKernel`, or the enclosing NVTX op under `--no-cuda-api`; the flow `id` is a **numeric** per-rank namespace `_flow_id = rank┬Ę1e10 + correlationId` ŌĆö correlationId is only unique within a profile so merged ranks would cross-link, and Perfetto's legacy-JSON flow binding silently drops *string* ids; the GPU end binds to the kernel's **emitted (clamped) start** via `TraceWriter.kslice`, not the raw `k.start`, or same-stream-overlapping kernels' flows land outside the slice), **`--comm-flows`/`--no-comm-flows`** (**cross-rank** communication flows, default on ŌĆö NCCL P2P and collectives are linked across ranks purely by their `P2p:commId=ŌĆ”:seq=` / `Op:comm=ŌĆ”:seq=` NVTX marker, which is globally consistent across the participants; the marker is used directly (no kernel association), endpoints on the issuing CPU threads, `cat="comm"`, id range `5e14+` distinct from launch flows; only drawn when Ōēź2 endpoints are in the trace ŌĆö click a send to jump to the matching recv on the peer rank), `--min-dur-ns NS` (drop short events to shrink). `-o OUT` writes `.json`/`.json.gz` (gz recommended); omit to just print the step table. | `gpu-exporter <db.sqlite>... [--step-nvtx SUBSTR] [--skip-steps N] [--step N] [--project] [--stack-depth N] [--no-align] [--cpu-nvtx] [--no-cpu-nvtx] [--no-cuda-api] [--no-flows] [--no-comm-flows] [--min-dur-ns NS] -o OUT]` |
| `gpu-sched` | **Config-only** (no profile) ŌĆö draw an **op-level 1F1B pipeline timeline** as SVG, **anchored to a unit** (1 unit = the `--unit` phase's time, default `F` = attn_fwd; or `--unit-ms`). Unlike `gpu-groups --gbs` (unit-time F/B Gantt showing only schedule *shape*), here every forward/backward op of every microbatch is **decomposed into its measured GPU sub-phases**, **each box as wide as its value** and **labelled with that value in units** (`F 1.00`, `D 1.33`, `Eß┤░ 1.36`, `L 7.90`, ŌĆ”; rotated to fit narrow boxes). Schedule order = Megatron's interleaved 1F1B (reuses `parallel_groups._pp_program` + `PPLayout`); op start times come from data dependencies + per-device durations (`schedule_timed`), so the diagram shows the true fill/steady/drain and **each pp rank's own bubble %** (printed on its row ŌĆö pp ranks holding light chunks like the embedding-only stage idle more). Sub-phases (one colored box each): forward **F** attn / **D** dispatch(EP) / **E** moe-mlp (router folded in) / **C** combine(EP) / **M** dense-mlp / **V** embedding / **L** lm-head (loss folded in); backward **Fß┤░/FßĄé**, **Eß┤░/EßĄé**, **Mß┤░/MßĄé** (d/w-grad), **D╩Ė** dispatch-bwd, **C╩Ė** combine-bwd, **V╩Ė**, **L╩Ė**. **`--ep-overlap` (default on; `--no-ep-overlap` for the serial single-lane view)** uses the **MoE EP-A2A overlap model** (Megatron `combined_1f1b.py`): each stage splits into **two lanes ŌĆö compute (attn/mlp) over comm (dispatch/combine A2A)** ŌĆö each MoE layer is 4 nodes on 2 in-order CUDA streams (`comp: attnŌåÆmlp`, `comm: dispatchŌåÆcombine`), and in steady 1F1B a forward mb is paired with an **independent** backward mb so the forward's A2A hides behind the backward's compute (and vice versa); same-mb F/B pairs (top stage) stay sequential (data-dependent). The two streams are list-scheduled by cross-stream data deps (`schedule_overlap` ŌåÆ `render_overlap_svg`), so hidden A2A sits *under* a compute box and exposed A2A (warmup/cooldown) sits *over* a compute gap; bubble is measured on the comp lane, and the header reports the serial-vs-overlap makespan saving. A `t` layer is MoE (F,D,E,C) unless its global id is in `--dense-layers` (F,M; default only layer 0). **`--recompute SPEC`** models fine-grained activation recomputation: it prepends a re-run of the forward to a layer's backward, drawn as **hatched F╩│/E╩│/C╩│/D╩│/M╩│ boxes** (`recompute_tokens` ŌåÆ nodes on the same comp/comm streams, chained so a MoE mlp re-forward's dispatch/combine A2A stays **exposed** on the comm lane, not hidden). SPEC picks **which layers re-run which parts**: `TERM(;TERM)*`, `TERM = LAYERS[:PARTS]`, `LAYERS = all | id | a-b` (comma list, clamped to the model), `PARTS = all | attn | mlp` (default both) ŌĆö e.g. `all:attn,mlp` (full), `all:attn` (attention-only selective), `all:mlp`, `0-20:mlp;30,32,34:attn`; default `none` (off). attnŌåÆre-run F; mlpŌåÆD,E,C (MoE) or M (dense). Runs through both `schedule_overlap` (EP-overlap) and `schedule_timed` (serial); the header prints the recomputed layer/part counts. **Per-phase times are params/config, no CSV**: built-in `DEFAULTS` (a measured A3B-8EP-4PP run; router folded into E, loss into L/LßĄé) overridden by a **`--config` JSON** whose keys are the phase tokens (`{"F":1.4,"E":1.59,"E^D":1.9}`) and/or the per-phase **`--t-<phase>` flags** (`--t-F`/`--t-E`/`--t-Fd`/`--t-Ew`/`--t-V`/`--t-L`/ŌĆ”); precedence `--t-*` > `--config` > default. `--dump-config` prints the effective times as a ready-to-edit JSON (unknown tokens are rejected). `--microbatches`/`--mbs N` (pipeline depth), `--px-per-unit` (h-scale), `--pp`, `--pipeline-model-parallel-layout`/`--pp-layout`. Colors are `hsl()` (browser-only; cairosvg can't rasterize hsl ŌĆö convert to hex first if you need a PNG). | `gpu-sched --pp N --pipeline-model-parallel-layout "DSL" [--microbatches N] [--unit PHASE] [--ep-overlap] [--dense-layers IDS] [--recompute SPEC] [--config times.json] [--t-F MS ŌĆ”] [--dump-config] [--px-per-unit PX] --svg OUT` |
| `gpu-groups` | **Config-only** (no profile) ŌĆö resolve the per-rank parallel process groups the way Megatron-LM lays them out, so for any rank you can see which ranks it talks to under each communication kind (TP / SP / CP / DP / PP / EP). ## Conventions

- `utils.common.open_db(path)` opens SQLite read-only with `sqlite3.Row` factory.
- `utils.kernel.KERNEL_SQL` is the canonical kernel+launch-API join; append `WHERE ŌĆ”` / `ORDER BY ŌĆ”` per query.
- `utils.nvtx.NvtxIndex`: `matches(substr)` finds frames by name (step partitioning); `iter_stacks(api_intervals)` bulk-resolves each kernel's enclosing stack via sweep-line.
- `utils.comm.load_comm_in_window(conn, idx, ws, we, dtype_bytes=2)` returns one `CommEvent(op, comm, seq, algo, bytes, peer, gpu_dur, gpu_start, scope)` per comm kernel that starts in `[ws, we]` ŌĆö both `ncclDevKernel` (op/PG-handle/algo/bytes/peer from the `nccl:<op>` range + `comm=` marker) and **DeepEP** MoE all-to-all (`deep_ep::{intra,inter}node::{dispatch,combine,notify,ŌĆ”}`; op `Dispatch`/`Combine`, `comm=None`, `algo` = topology, volume from the enclosing `FusedDispatch`/`FusedCombine` range ├Ś `dtype_bytes`). `scope` is the issuing NVTX scope from `_scope_from_stack` (innermost non-plumbing frame, else the outermost frame). The caller builds the `NvtxIndex` + step window and passes them in (kernel-first, NVTX via the mapping). The basis for `gpu-comm`.
- CLI tools live in `tools/`, keep an `if __name__ == "__main__":` block, use package-relative imports, and get a `gpu-*` console entry point via `cli.py` + `[project.scripts]`. The flame-graph renderer (StackNode / write_html) lives in `flamegraph.py`; both `gpu_comm.py` and `kernel_shapes.py` reuse only its step-window helpers (`detect_steps` / `compute_step_windows`; `kernel_shapes.py` also reuses `normalize` + `KERNELS_IN_WINDOW_SQL`), loads ranks in parallel via `load_all` (`ProcessPoolExecutor`), and prints three scope-keyed sections (`_print_collectives` / `_print_p2p` / `_print_ep`). The heatmap renderer (`render_heatmap` in `gpu_comm.py`) is matplotlib-only and kept off the data path.
