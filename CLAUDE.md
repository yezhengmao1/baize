# CLAUDE.md

Working directory for the **nvidia-nsys-analyzer** agent. Python tools for analyzing NVIDIA Nsight Systems profiling data (`.nsys-rep` ‚Üí SQLite).

**Status**: Installable package `baize` with five console entry points ‚Äî `gpu-flame` (per-step kernel‚ÜíNVTX flame graph, incl. `--diff`), `gpu-comm` (cross-rank communication ‚Äî NCCL collectives/P2P + DeepEP MoE all-to-all: volume / busbw / skew), `gpu-shape` (per-operator input-shape table, compute only), `gpu-exporter` (Chrome/Perfetto trace JSON), and `gpu-groups` (Megatron parallel-group resolver, config-only ‚Äî no profile). (The earlier suite `perf_stat.py`, `kernel_analysis.py`, `comm_bw.py`, ‚Ä¶ and the standalone `report.py`/`kernel_inspect.py`/`comm_inspect.py` were removed.)

**Data model rule**: kernels are the primary entity; NVTX context is attached via the kernel->NVTX mapping (`utils.nvtx.NvtxIndex`, or `utils.comm` for the NCCL-kernel‚Üíop join). Tools may reverse-look-up kernels through this mapping but must not query `NVTX_EVENTS` as an independent first-class table.

## Environment

- **nsys**: `/usr/local/bin/nsys` (2026.2.1); convert with `nsys export --type sqlite -o out.sqlite in.nsys-rep`.
- **Python**: 3.12. Runtime deps: `numpy`, `matplotlib` (declared in `pyproject.toml`); `sqlite3` is stdlib.

## Install

```bash
pip install -e .          # from repo root; provides gpu-flame / gpu-comm / gpu-shape / gpu-exporter / gpu-groups
```

Entry points run the tools' `if __name__ == "__main__":` blocks via small shims in `nsys_tools/cli.py` (so the CLI modules keep that style). Equivalent: `python -m nsys_tools.tools.flamegraph ‚Ä¶`.

## Tools

| Binary | Purpose | Usage |
|---|---|---|
| `gpu-flame` | Step-boundary detector + optional flame graph. Detects step boundaries from the kernel->NVTX mapping (`--step-nvtx SUBSTR`, default `Optimizer.step`), skips warmup (`--skip-steps N`, default 1), prints the per-step window table. With `--flamegraph OUT` builds an interactive HTML flame graph (`<OUT>.html`, d3-flame-graph) with **four charts over one shared hierarchy**: **Solo** (non-overlap; widths = Œ£ per-activity solo time / `active_count==1`, the portion blocking the GPU clock), **Sum** (overlap; widths = Œ£ GPU-activity time, color saturation encodes overlap: vivid = exposed, pale = hidden), **Count** (widths = number of GPU activities / kernel launches ‚Äî wide here but narrow in Sum ‚áí launch-bound, a fusion/CUDA-graph target), and **Sum + idle** (the Sum chart with GPU-idle added). `sum‚Üísolo` shrink = overlapped/hidden, equal = exposed; every node carries `sum`/`solo`/`count` (StackNode.count) and the tooltip shows time (sum/solo/overlap%) + count/% of launches. `--stack-depth N` truncates the CPU NVTX path. Leaves are compute kernels **plus GPU memcpy (H2D/D2H/D2D) and memset** (`--include-memcpy`/`--no-include-memcpy`, default on) ‚Äî these live in the separate `CUPTI_ACTIVITY_KIND_MEMCPY`/`_MEMSET` tables (joined to their launch API via `correlationId`, same NVTX mapping as kernels), colored green/purple; with them off, memcpy-only spans fall into idle. **GPU idle appears only in the third chart**: gaps are attributed to the NVTX scope enclosing each gap (`--attribute-idle`/`--no-attribute-idle`, default on) ‚Äî magenta `<idle>` leaves so you can see *which phase* the GPU stalled in (gaps resolved via the same sweep-line as kernels, keyed on GPU wall-time ‚Üí approximate under heavy CPU-ahead async); `--no-attribute-idle` makes it one root block. The Solo/Sum charts are pure GPU activity (no idle). Data model: `load_flame_tree` returns `(tree, tree_idle)` ‚Äî `tree_idle` is `tree.clone()` + idle; `write_html` emits both as `__DATA__`/`__DATA_IDLE__`. **`--diff BASELINE_DB`** emits a differential flame graph instead: `db` is the "after", BASELINE the "before", values per-step-normalized, each frame colored by delta (red = more GPU time now, blue = less), two views = before/after widths so removed and added frames are both visible (renderer: `write_diff_html` + `templates/flamegraph_diff.html`). | `gpu-flame <db.sqlite> [--step-nvtx SUBSTR] [--skip-steps N] [--stack-depth N] [--no-include-memcpy] [--no-attribute-idle] [--flamegraph OUT] [--diff BASELINE_DB]` |
| `gpu-shape` | Per-operator input-shape table over the post-warmup window, **compute only** (communication excluded). Reuses `gpu-flame`'s step detector (`--step-nvtx`/`--skip-steps`) to pick the window, reads every kernel via `KERNEL_SQL`, and attributes each to its **innermost enclosing aten-op NVTX frame** ‚Äî the frame carrying a `sizes = [[...]]` shape annotation (torch `emit_nvtx(record_shapes=True)`), resolved through the kernel‚ÜíNVTX-stack mapping. `op_frame` skips comm-plumbing frames (`nccl`/`c10d::`/`record_param_comms`/`NcclGroup`/`deep_ep`/`Fused{Dispatch,Combine}`) and `ncclDevKernel*` kernels are dropped outright, so the table is pure compute. One op **invocation** = one op-frame instance `(op, shapes, start, end)`; instances roll up per **(operator, input-shapes)** ‚Üí `calls` (invocations), `kern` (kernels), Œ£`gpu` time, `avg/call`, `%` of compute GPU time. Header reports excluded comm / no-shape kernel counts+time so coverage is explicit. `--sort {time,calls,op}`, `--top N`, `--csv OUT` (full untruncated rows). **`--html OUT`** writes a self-contained interactive page (`write_shape_html` + `templates/shape_table.html`, no CDN; styled to match `flamegraph.html`): a summary note (compute Œ£ / #ops / kernels / excluded comm & no-shape), ranked horizontal bars of the top time sinks and a sortable/filterable table, both driven by a shared grouping toggle ‚Äî **by (op,shapes)** / **by operator** / **by kernel** ‚Äî all compute-only. **by kernel** aggregates every compute kernel by `shortName` (`build_shape_table` also returns `kern_names`; **all communication excluded** ‚Äî NCCL by name via `_COMM_KERNEL_PREFIX`, DeepEP `dispatch`/`combine` etc. by their enclosing comm NVTX scope via `_stack_has_comm`), including the kernels the op table drops as no-shape; its table columns switch to launches/Œ£gpu/avg/min/max. Each bar colored by name via the same fnv1a-hash‚ÜíHSL scheme as the flame graph; hover = full shapes / per-call & min-max stats. | `gpu-shape <db.sqlite> [--step-nvtx SUBSTR] [--skip-steps N] [--sort {time,calls,op}] [--top N] [--csv OUT] [--html OUT]` |
| `gpu-comm` | Cross-rank communication report over many ranks. Loads each rank's comm kernels **in parallel** (`--jobs N`, multiprocessing; ranks are independent files). Every comm row is keyed by its **issuing NVTX scope** ‚Äî the model phase / call site that issued it (e.g. `_LinearBackward`, `SinkCorrectionWithCPFunc`, or a bare `c10d::allreduce_coalesced_` / `c10d::send` when no model module wraps it), resolved via the kernel‚ÜíNVTX-stack mapping and **falling back to the outermost frame** when every enclosing frame is comm plumbing (`nccl`/`c10d::`/`record_param_comms`/`NcclGroup`). Three sections: **(1) Collectives** (`ncclDevKernel` AllGather/ReduceScatter/AllReduce/Broadcast): aligned across ranks by **(comm-handle, op, seq)** ‚Äî the `comm=0x‚Ä¶` handle is the real PG identity (globally consistent across the ranks sharing the group), so each collective instance's per-rank durations and true group size are recovered exactly (no traffic-signature heuristic), then rolled up per (scope, op, width) ‚Üí `nranks` (real group width, e.g. 8/32/96), per-rank `calls`, Œ£`vol/rank` (per rank, √∑ group count), **busbw@floor** (`factor(op,nranks)√óbytes / min-time-across-ranks`, busy-wait removed ‚Üí achievable BW), **wait%** (`Œ£(time‚àífloor)/Œ£time`, floor = min within each collective instance) + worst rank. **(2) P2P send/recv**: peer-based, no `comm=`/seq ‚Üí no clean group, so only group-independent quantities per (scope, op): Œ£`vol/rank`, `calls/rank`, per-rank GPU-time floor..max. **(3) EP all-to-all (DeepEP)** (`deep_ep::` dispatch/combine, incl. notify/layout helpers; volume from `FusedDispatch`/`FusedCombine` NVTX sizes): per (scope, phase), skew computed **within EP groups** of `--ep N` consecutive ranks (`rank//N`, default 8 = intranode) ‚Äî a flat all-ranks floor would conflate the independent groups. `--p2p-dtype-bytes N` sizes the marker-less P2P/DeepEP volume (default 2 = bf16; 1 = fp8, 4 = fp32). `--heatmap OUT` writes a per-rank√ócomm skew PNG (`--by-node` averages rows by node; `--sort {slowness,id}`). | `gpu-comm <db.sqlite>... [--step-nvtx SUBSTR] [--skip-steps N] [--p2p-dtype-bytes N] [--ep N] [--heatmap OUT] [--by-node] [--sort {slowness,id}] [--jobs N]` |
| `gpu-exporter` | Export a step (or the whole post-warmup window) as **Chrome Trace Event Format JSON** ‚Äî what `chrome://tracing` and the Perfetto UI ingest (nsys itself has no such export). Reuses `gpu-flame`'s step detector (`--step-nvtx`/`--skip-steps`; `--step N` picks one 1-based step, else the full post-warmup window). Default **split view**: GPU streams carry kernels/memcpy/memset, NVTX ranges stay on the CPU threads that issued them; `--project` instead resolves each GPU op's enclosing NVTX stack (via the kernel‚ÜíNVTX launch-API mapping) and nests it onto the op's GPU stream so every kernel sits inside its issuing scopes (kernel = leaf), `--stack-depth N` caps that nesting. Multiple ranks merge into one trace (each rank its own pid namespace, `R<rank>` track prefix) **wall-clock aligned by each file's session-start UTC epoch** so cross-node ranks land on one real timeline and skew is visible (`--no-align` overlays at t=0). Extras: `--cpu-nvtx`/`--no-cpu-nvtx`, **`--cuda-api`/`--no-cuda-api`** (CUDA runtime API ‚Äî `cudaLaunchKernel`, ‚Ä¶ ‚Äî on CPU threads, **default on**, heavy), **`--flows`/`--no-flows`** (CPU-launch‚ÜíGPU-kernel flow arrows, **default on** ‚Äî click a GPU kernel in Perfetto to jump to its launch site; both endpoints use `bp:"e"` to bind the enclosing slice ‚Äî the `cudaLaunchKernel`, or the enclosing NVTX op under `--no-cuda-api`; the flow `id` is a **numeric** per-rank namespace `_flow_id = rank¬∑1e10 + correlationId` ‚Äî correlationId is only unique within a profile so merged ranks would cross-link, and Perfetto's legacy-JSON flow binding silently drops *string* ids; the GPU end binds to the kernel's **emitted (clamped) start** via `TraceWriter.kslice`, not the raw `k.start`, or same-stream-overlapping kernels' flows land outside the slice), `--min-dur-ns NS` (drop short events to shrink). `-o OUT` writes `.json`/`.json.gz` (gz recommended); omit to just print the step table. | `gpu-exporter <db.sqlite>... [--step-nvtx SUBSTR] [--skip-steps N] [--step N] [--project] [--stack-depth N] [--no-align] [--cpu-nvtx] [--no-cpu-nvtx] [--no-cuda-api] [--no-flows] [--min-dur-ns NS] -o OUT]` |
| `gpu-groups` | **Config-only** (no profile) ‚Äî resolve the per-rank parallel process groups the way Megatron-LM lays them out, so for any rank you can see which ranks it talks to under each communication kind (TP / SP / CP / DP / PP / EP). Reimplements Megatron's `RankGenerator` / `generate_masked_orthogonal_rank_groups`: global rank = mixed-radix over the axes in `--order` (default `tp-sp-cp-ep-dp-pp`); each group = ranks differing only along that axis. Two generators (decoder: tp,sp,cp,dp,pp; expert/MoE: tp(=etp),sp,ep,dp(=edp),pp). Sizes via `--tp/--sp/--cp/--pp/--dp/--ep/--etp/--world`; `--sp 1` = Megatron semantics (SP rides TP), `--sp N>1` = own axis. Pipeline layer layout via `--pipeline-model-parallel-layout`/`--pp-layout` (Megatron DSL, vpp inferred). `--rank R` focuses one rank, `--limit N`, `--csv OUT` (full per-rank), `--svg OUT` (node-organized DP data-flow diagram), `--gpus-per-node N`. **`--gbs N` [`--mbs N`]** appends a **1F1B pipeline-schedule Gantt** for one pp group to the `--svg` (#microbatches = gbs/(dp¬∑mbs); `pp_schedule` runs Megatron's fixed per-stage 1F1B order ‚Äî interleaved when vpp>1 ‚Äî and times it by data deps: bubble = (pp‚àí1)/(vpp¬∑#microbatches), forward=blue / backward=orange, darker = later vpp chunk). | `gpu-groups --world N --tp N --pp N --ep N [--sp N] [--cp N] [--etp N] [--order A-B-‚Ä¶] [--pp-layout ‚Ä¶] [--rank R] [--csv OUT] [--svg OUT] [--gbs N] [--mbs N]` |

Example:
```bash
gpu-flame profile.sqlite --flamegraph /tmp/flame
gpu-flame after.sqlite --diff before.sqlite --flamegraph /tmp/diff
gpu-comm rank*.sqlite --jobs 32 --ep 8 --heatmap /tmp/skew --by-node
gpu-shape profile.sqlite --top 30 --csv /tmp/shapes
gpu-exporter rank*.sqlite --step 3 -o /tmp/trace.json.gz   # open in ui.perfetto.dev
gpu-groups --world 64 --tp 2 --pp 4 --ep 8 --rank 0 --svg /tmp/groups
```

## Key SQLite Tables

| Table | Contents |
|---|---|
| `CUPTI_ACTIVITY_KIND_KERNEL` | GPU kernels (start, end, shortName, grid/block dims, regs, sharedMem, correlationId) |
| `CUPTI_ACTIVITY_KIND_RUNTIME` | CUDA API calls (nameId, start, end, correlationId) ‚Äî joins kernels to launch site |
| `NVTX_EVENTS` | NVTX markers (textId or inline text, start, end, globalTid) |
| `StringIds` | String lookup (id ‚Üí value) |
| `TARGET_INFO_SYSTEM_ENV` | `DeviceEnvironment` env dump ‚Äî `RANK=` (global rank), `HOSTNAME=` (node) |

## Conventions

- `utils.common.open_db(path)` opens SQLite read-only with `sqlite3.Row` factory.
- `utils.kernel.KERNEL_SQL` is the canonical kernel+launch-API join; append `WHERE ‚Ä¶` / `ORDER BY ‚Ä¶` per query.
- `utils.nvtx.NvtxIndex`: `matches(substr)` finds frames by name (step partitioning); `iter_stacks(api_intervals)` bulk-resolves each kernel's enclosing stack via sweep-line.
- `utils.comm.load_comm_in_window(conn, idx, ws, we, dtype_bytes=2)` returns one `CommEvent(op, comm, seq, algo, bytes, peer, gpu_dur, gpu_start, scope)` per comm kernel that starts in `[ws, we]` ‚Äî both `ncclDevKernel` (op/PG-handle/algo/bytes/peer from the `nccl:<op>` range + `comm=` marker) and **DeepEP** MoE all-to-all (`deep_ep::{intra,inter}node::{dispatch,combine,notify,‚Ä¶}`; op `Dispatch`/`Combine`, `comm=None`, `algo` = topology, volume from the enclosing `FusedDispatch`/`FusedCombine` range √ó `dtype_bytes`). `scope` is the issuing NVTX scope from `_scope_from_stack` (innermost non-plumbing frame, else the outermost frame). The caller builds the `NvtxIndex` + step window and passes them in (kernel-first, NVTX via the mapping). The basis for `gpu-comm`.
- CLI tools live in `tools/`, keep an `if __name__ == "__main__":` block, use package-relative imports, and get a `gpu-*` console entry point via `cli.py` + `[project.scripts]`. The flame-graph renderer (StackNode / write_html) lives in `flamegraph.py`; both `gpu_comm.py` and `kernel_shapes.py` reuse only its step-window helpers (`detect_steps` / `compute_step_windows`; `kernel_shapes.py` also reuses `normalize` + `KERNELS_IN_WINDOW_SQL`), loads ranks in parallel via `load_all` (`ProcessPoolExecutor`), and prints three scope-keyed sections (`_print_collectives` / `_print_p2p` / `_print_ep`). The heatmap renderer (`render_heatmap` in `gpu_comm.py`) is matplotlib-only and kept off the data path.
